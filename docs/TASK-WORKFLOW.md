## 📋 一、任务生命周期

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐

└─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘
                   ↑              ↑              ↑
                   │              │              │
              ┌────┴────┐   ┌────┴────┐   ┌────┴────┐

              └─────────┘   └─────────┘   └─────────┘
```

## 🎯 二、角色职责与工作顺序

### 1️⃣ 小策📋 (产品经理) - 第一步
**职责**: 需求分析与产品设计

| 产出物 | 存放路径 | 格式要求 |
|--------|----------|----------|
| 产品需求文档 | `docs/prd/TASK-XXX-prd.md` | Markdown |
| 用户故事 | `docs/prd/TASK-XXX-stories.md` | Markdown |
| 功能清单 | `docs/prd/TASK-XXX-features.md` | Markdown |

**完成标准**:
- [ ] PRD 文档完整
- [ ] 功能清单明确
- [ ] 用户故事清晰
- [ ] 提交 Git 并通知小管

---

### 2️⃣ 小艺🎨 (设计师) - 第二步
**职责**: 界面设计与交互设计

| 产出物 | 存放路径 | 格式要求 |
|--------|----------|----------|
| 设计稿 | `docs/design/TASK-XXX-design.md` | Markdown + 图片 |
| 交互原型 | `deliverables/code/prototype/` | HTML/可交互文件 |
| 设计规范 | `docs/design/TASK-XXX-spec.md` | Markdown |

**完成标准**:
- [ ] 设计稿完整
- [ ] 交互原型可预览
- [ ] 设计规范明确
- [ ] 提交 Git 并通知小管

---

### 3️⃣ 小管📊 (PMO) - 评审确认
**职责**: 质量把关与流程控制

| 检查项 | 检查内容 | 确认方式 |
|--------|----------|----------|
| PRD 评审 | 需求是否完整、清晰 | `docs/review/TASK-XXX-prd-review.md` |
| 设计评审 | 设计是否符合需求 | `docs/review/TASK-XXX-design-review.md` |
| 准入确认 | 是否允许进入开发 | 更新任务状态为 `dev_ready` |

**完成标准**:
- [ ] PRD 评审通过
- [ ] 设计评审通过
- [ ] 签署准入确认
- [ ] 通知小码可以开始开发

---

### 4️⃣ 小码💻 (开发) - 第三步
**职责**: 代码实现与单元测试

| 产出物 | 存放路径 | 格式要求 |
|--------|----------|----------|
| 源代码 | `deliverables/code/TASK-XXX/` | 按项目规范 |
| 数据库设计 | `deliverables/code/TASK-XXX/db/` | SQL/Schema |
| API 文档 | `docs/specs/TASK-XXX-api.md` | Markdown |
| 单元测试 | `deliverables/test/TASK-XXX/` | 测试脚本 |

**完成标准**:
- [ ] 代码完成并提交
- [ ] 单元测试通过
- [ ] API 文档完整
- [ ] 提交 Git 并通知小管

---

### 5️⃣ 小测🧪 (测试) - 第四步
**职责**: 功能测试与质量验证

| 产出物 | 存放路径 | 格式要求 |
|--------|----------|----------|
| 测试用例 | `deliverables/test/TASK-XXX-cases.md` | Markdown |
| 测试报告 | `deliverables/review/TASK-XXX-test-report.md` | Markdown |
| Bug 列表 | `deliverables/review/TASK-XXX-bugs.md` | Markdown |

**完成标准**:
- [ ] 测试用例执行完毕
- [ ] 测试报告提交
- [ ] 严重 Bug 已修复
- [ ] 通知小管验收

---

### 6️⃣ 小管📊 (PMO) - 最终验收
**职责**: 最终质量确认与任务关闭

| 检查项 | 检查内容 | 确认方式 |
|--------|----------|----------|
| 代码验收 | 代码质量、规范 | Code Review |
| 功能验收 | 是否满足 PRD | 功能演示 |
| 文档验收 | 文档完整性 | 文档检查 |
| 任务关闭 | 签署完成确认 | 更新任务状态为 `done` |

## 📁 三、产物提交规范

### 目录结构
```
~/.openclaw/workspace/
├── docs/
│   ├── prd/           # 产品需求文档 (小策)
│   │   └── TASK-XXX-prd.md
│   ├── design/        # 设计文档 (小艺)
│   │   └── TASK-XXX-design.md
│   ├── specs/         # 技术规格 (小码)
│   │   └── TASK-XXX-api.md
│   └── review/        # 评审记录 (小管)
│       └── TASK-XXX-xxx-review.md
│
├── deliverables/
│   ├── code/          # 源代码 (小码)
│   │   └── TASK-XXX/
│   ├── test/          # 测试文件 (小测)
│   │   └── TASK-XXX/
│   └── review/        # 验收报告 (小管)
│       └── TASK-XXX-final-review.md
│
└── .git/              # 版本控制
```

### Git 提交规范
```bash
# 小策 - 产品文档
git commit -m "feat(prd): TASK-001 产品需求文档"

# 小艺 - 设计文档
git commit -m "feat(design): TASK-001 界面设计稿"

# 小管 - 评审确认
git commit -m "review(prd): TASK-001 PRD 评审通过"

# 小码 - 代码提交
git commit -m "feat(dev): TASK-001 后端 API 实现"

# 小测 - 测试报告
git commit -m "test: TASK-001 功能测试报告"
```

## ⚠️ 四、流程约束规则

### 🚫 禁止行为
1. **禁止跳过评审** - 小管未确认前，不得进入下一步
2. **禁止无文档开发** - 没有 PRD/设计稿，小码不得开工
3. **禁止无测试上线** - 没有测试报告，小管不得验收
4. **禁止延迟提交** - 每个阶段完成后 24 小时内必须提交产物

### ✅ 强制检查点
| 检查点 | 负责人 | 检查内容 | 通过标准 |
|--------|--------|----------|----------|
| CP1 | 小管 | PRD 完整性 | 需求清晰、功能完整 |
| CP2 | 小管 | 设计符合性 | 设计满足 PRD 要求 |
| CP3 | 小管 | 开发准入 | PRD+ 设计均通过 |
| CP4 | 小管 | 代码质量 | Code Review 通过 |
| CP5 | 小管 | 测试覆盖 | 测试报告完整 |
| CP6 | 小管 | 最终验收 | 所有产出物完整 |

## 📊 五、任务状态定义

| 状态 | 英文 | 说明 | 下一步 |
|------|------|------|--------|
| 待办 | `todo` | 任务已创建，等待开始 | → 设计中 |
| 设计中 | `designing` | 小策/小艺正在工作 | → 评审中 |
| 评审中 | `reviewing` | 小管正在评审 | → 开发中/打回 |
| 开发中 | `developing` | 小码正在编码 | → 测试中 |
| 测试中 | `testing` | 小测正在测试 | → 验收中 |
| 验收中 | `verifying` | 小管最终验收 | → 已完成/打回 |
| 已完成 | `done` | 任务完成 | - |
| 已阻塞 | `blocked` | 遇到问题，等待解决 | → 待办/设计中 |

## 🎯 六、绩效挂钩

### 积分规则
| 行为 | 积分 | 说明 |
|------|------|------|
| 按时提交产物 | +5 | 每个阶段 |
| 提前完成 | +2 | 每提前 1 小时 |
| 质量优秀 | +3 | 小管评定 |
| 延迟提交 | -2 | 每延迟 1 小时 |
| 跳过流程 | -10 | 严重违规 |
| 产物不合格 | -5 | 被打回重做 |

### 绩效评级
| 评级 | 分数范围 | 说明 |
|------|----------|------|
| S | 95+ | 卓越 |
| A | 85-94 | 优秀 |
| B | 70-84 | 良好 |
| C | 60-69 | 合格 |
| D | <60 | 需改进 |

## 📞 八、沟通机制

### 每日站会
- **时间**: 每天 09:00
- **参与**: 全员
- **内容**: 昨日进度、今日计划、遇到的问题

### 阶段汇报
- **触发**: 每个阶段完成时
- **汇报人**: 当前阶段负责人
- **接收人**: 小管 + 虾管事

### 风险预警
- **触发**: 发现可能延期/质量问题
- **汇报人**: 发现问题的人
- **接收人**: 小管 + 虾管事
